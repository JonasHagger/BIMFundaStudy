Chapter 3 — Data Exchange & IFC

The vendor-neutral exchange standard. IFC carries geometry and semantics in a layered, object-oriented schema (built on STEP/EXPRESS), with objectified relationships and multiple geometry representations. This is the single heaviest exam topic.

Tags say what a card is. Green box = the direct exam answer.

Concept cards

IFC — the core facts

DefinitionExamCard 1

Exam: DE-Q24 (ISO standard) · DE-Q27 (IFC model) · DE-Q29 (why vendor-neutral) · DE-Q30 (purpose)

IFC = Industry Foundation Classes — the standardized data model for exchanging BIM, carrying geometry AND semantics. Developed by buildingSMART (non-profit, 1995).

IFC's own standard ISO 16739

Built on STEP = ISO 10303 (inherits EXPRESS + P21 file)

Why vendor-neutral? / purpose of exchange

Fair vendor competition, no lock-in, public bodies can't demand one vendor. Purpose: handover to client · coordinate disciplines · hand model to next phase.

Exam answer"IFC based on which ISO standard?" → STEP, ISO 10303 (IFC itself is ISO 16739). IFC = vendor-neutral model for geometry + semantics by buildingSMART.

EXPRESS, STEP & geometry-only formats

DefinitionExamCard 2

Exam: DE-Q25 (schema language) · DE-Q26 (geometry-only formats)

EXPRESS the data-modeling language that defines the IFC schema (EXPRESS-G = graphical).

STEP ISO 10303, "Exchange of Product model data" = geometry + semantics; file = STEP-P21 (.ifc).

Geometry-only formats (no semantics)

STLVRMLOBJDXF/IGES
Exam answerSchema language = EXPRESS. Geometry-only = STL, VRML, OBJ, IGES → no downstream use.

IFC layered architecture + IfcRoot

ModelExamCard 3

Exam: DE-Q32, DE-Q33 · MP-Q2 (order) · MP-Q3 (which layer ≠ IfcRoot) — asked every year

Most general (bottom) → most specific (top)

Exam answerOrder general→specific: Resource → Core → Interoperability → Domain. The Resource layer does NOT inherit from IfcRoot (no GUID/identity). Most entities derive from IfcRoot, which sits in the Core layer.

Schema vs Instance

DefinitionExamCard 4

Exam: DE-Q31 · MP-Q10 — schema vs instance in IFC & OOP

Schema the data model: class hierarchy + properties (defined in EXPRESS). The "blueprint".

Instance an actual object of a class = the concrete building model = the .ifc file (STEP-P21).

In Revit/ArchiCAD: implemented as a catalogue of parametric object types with geometric + semantic attributes.

Exam answerSchema = class definitions (blueprint); instance = a concrete object of that class (the .ifc file).

Model View Definition (MVD)

DefinitionExamCard 5

Exam: DE-Q36, DE-Q65 (MVD) · MP-Q14 (mvdXML true/false)

A subset of the full IFC schema defining mandatory + optional objects/attributes for a specific exchange/use case. Enables software certification. mvdXML = its machine-readable definition.

IFC2x3 Coordination View, Structural Analysis View, FM Handover

IFC4 Reference View, Design Transfer View

Exam answerMVD = subset of the IFC schema for a specific exchange scenario (e.g. IFC4 Reference View). Comes from the IDM (Ch 10).

Objectified relationships (wall · opening · window)

SketchExamCard 6

Exam: DE-Q37 — explain with a wall, window, opening — asked repeatedly

Relationships are their OWN objects (reified), not just attributes

Exam answerThe wall is voided by an opening via IfcRelVoidsElement; the opening is filled by the window via IfcRelFillsElement. Reifying the relationship lets it carry its own attributes and n-ary links.

IfcBuildingElementProxy

DefinitionExamCard 8

Exam: DE-Q34 — what it is + one example

A proxy with the same function as any IfcBuiltElement subtype, but without a predefined element meaning. Used for elements not (yet) defined in IFC. Example: bridge bearing; heavily used in infrastructure.

Exam answerPlaceholder for undefined element types (e.g. bridge bearing). Note: unknown Revit families export as proxies by default.

IFC: unique features & the "wrong" statement

MC trapExamCard 9

Exam: DE-Q28 (which fact is wrong) · DE-Q31 (unique features)

Unique features (vs STL/DWG/OBJ)

Exchanges both semantics + geometry; allows several geometry representations per semantic object.

Watch out — the FALSE statement: "Property Sets can only store info that is part of the standardized IFC schema." That's wrong — Psets store info OUTSIDE the schema (name-value pairs, no schema extension needed).

Exam answerWrong statement = the one restricting Property Sets to the standardized schema. True: geometry/semantics separated, IFC works for infrastructure, several geometry reps allowed.
